# 大文件断点续传下载方案对比

## 概述

大文件下载时可能因网络中断、浏览器刷新、服务端重启等原因失败，需要支持断点续传功能，即从上次下载中断的位置继续下载，而非重新开始。

---

## 方案一：HTTP Range 请求（推荐）

### 原理

使用 HTTP 标准协议中的 `Range` 和 `Content-Range` 头部，客户端请求文件的特定字节范围，服务端返回对应数据。

### 流程图

```
客户端                              服务端
  │                                    │
  │  1. 首次请求（HEAD）               │
  │ ───────────────> GET /file/info    │
  │ <─────────────── 返回文件元数据    │
  │                    Accept-Ranges: bytes
  │                    Content-Length: 1GB
  │                                    │
  │  2. 分段下载（可并发）              │
  │ ───────────────> Range: bytes=0-5MB│
  │ <─────────────── 206 Partial Content
  │                  Content-Range: bytes 0-5MB/1GB
  │                                    │
  │  3. 断点续传（中断后）              │
  │ ───────────────> Range: bytes=50MB-│
  │ <─────────────── 从50MB位置返回数据 │
```

### 服务端实现

```csharp
// 支持 Range 的下载接口
[HttpGet("download/{versionId}")]
public async Task<IActionResult> DownloadRange(
    Guid versionId, 
    [FromHeader] string? range,
    CancellationToken cancellationToken)
{
    var version = await _versionService.GetVersionAsync(versionId);
    
    // 解析 Range 头部
    var (start, end) = ParseRange(range, version.Size);
    
    // 返回指定范围的数据
    var stream = await _storageProvider.GetRangeStreamAsync(
        version.ObjectName, start, end, cancellationToken);
    
    Response.StatusCode = 206;
    Response.Headers.Append("Content-Range", $"bytes {start}-{end}/{version.Size}");
    
    return File(stream, "application/octet-stream", enableRangeProcessing: true);
}
```

### 客户端实现（JavaScript）

```javascript
class ResumableDownloader {
  constructor(url, fileName) {
    this.url = url;
    this.fileName = fileName;
    this.chunkSize = 5 * 1024 * 1024; // 5MB
    this.downloadedChunks = new Set();
  }

  async download() {
    // 获取文件信息
    const info = await fetch(`${this.url}/info`).then(r => r.json());
    
    // 检查本地存储的进度
    const savedProgress = localStorage.getItem(`dl_${this.fileName}`);
    if (savedProgress) {
      this.downloadedChunks = new Set(JSON.parse(savedProgress));
    }

    // 下载缺失的分片
    const chunks = [];
    for (let i = 0; i < info.totalChunks; i++) {
      if (!this.downloadedChunks.has(i)) {
        const chunk = await this.downloadChunk(i);
        chunks.push(chunk);
        this.downloadedChunks.add(i);
        
        // 保存进度
        localStorage.setItem(
          `dl_${this.fileName}`, 
          JSON.stringify([...this.downloadedChunks])
        );
      }
    }

    // 合并并保存
    this.mergeAndSave(chunks);
  }

  async downloadChunk(index) {
    const start = index * this.chunkSize;
    const end = start + this.chunkSize - 1;
    
    const response = await fetch(this.url, {
      headers: { 'Range': `bytes=${start}-${end}` }
    });
    
    return response.blob();
  }
}
```

### 优点

| 优点 | 说明 |
|------|------|
| 标准协议 | HTTP/1.1 原生支持，兼容性好 |
| 简单易实现 | 无需额外协议或格式 |
| 支持并发 | 可同时下载多个分片 |
| 服务端无状态 | 无需记录下载进度 |

### 缺点

| 缺点 | 说明 |
|------|------|
| 需要客户端配合 | 客户端需实现分片管理和合并 |
| 浏览器兼容性 | 旧版浏览器可能不支持 |
| 无进度持久化 | 服务端不记录进度，换设备需重传 |

## 方案二：预签名URL分段下载

### 原理

服务端生成多个预签名的分段下载URL，客户端直接使用这些URL下载，减轻服务端带宽压力。

### 流程图

```
客户端                              服务端
  │                                    │
  │  1. 请求分片计划                    │
  │ ───────────────> GET /chunks       │
  │                    {chunkSize: 5MB}│
  │ <─────────────── 返回分片列表      │
  │                    [               │
  │                      {             │
  │                        index: 0,   │
  │                        url: "presigned-url-0",
  │                        md5: "xxx"  │
  │                      },            │
  │                      ...           │
  │                    ]               │
  │                                    │
  │  2. 直接访问预签名URL下载            │
  │ ───────────────> GET presigned-url-0
  │ <───────────────────────────────────
  │     从对象存储直接返回数据           │
  │     （不经过应用服务器）             │
```

### 服务端实现

```csharp
[HttpGet("{versionId}/chunks")]
public async Task<IActionResult> GetChunkUrls(
    Guid versionId, 
    [FromQuery] long chunkSize = 5 * 1024 * 1024)
{
    var version = await _versionService.GetVersionAsync(versionId);
    var totalChunks = (int)Math.Ceiling((double)version.Size / chunkSize);
    
    var chunks = new List<ChunkInfo>();
    for (int i = 0; i < totalChunks; i++)
    {
        var start = i * chunkSize;
        var end = Math.Min(start + chunkSize - 1, version.Size - 1);
        
        // 生成预签名URL（MinIO/S3支持Range参数）
        var url = await _storageProvider.GetPresignedUrlAsync(
            version.ObjectName, 
            TimeSpan.FromHours(1),
            start,
            end);
        
        chunks.Add(new ChunkInfo
        {
            Index = i,
            Start = start,
            End = end,
            Url = url,
            Md5 = await CalculateChunkMd5Async(version.ObjectName, start, end)
        });
    }
    
    return Ok(new { fileSize = version.Size, chunks });
}
```

### 优点

| 优点 | 说明 |
|------|------|
| 减轻服务端压力 | 数据直接从对象存储返回 |
| 可控制访问 | URL有过期时间，安全性好 |
| 支持CDN | 预签名URL可使用CDN加速 |

### 缺点

| 缺点 | 说明 |
|------|------|
| 依赖存储特性 | MinIO/S3需支持预签名URL |
| URL管理复杂 | 大量分片时URL列表很大 |
| 无法统一控制 | 下载过程服务端无法干预 |

## 方案三：服务端代理流（带进度记录）

### 原理

所有下载流量经过服务端，服务端记录每个客户端的下载进度，支持精确的断点续传控制。

### 流程图

```
客户端                              服务端                              存储
  │                                    │                                   │
  │  1. 创建下载任务                    │                                   │
  │ ───────────────> POST /download    │                                   │
  │                    {versionId, clientId}
  │ <─────────────── 返回taskId        │                                   │
  │                                    │                                   │
  │  2. 开始下载（断点续传）            │                                   │
  │ ───────────────> GET /download/taskId
  │                    Range: bytes=50MB-
  │                                    │ ───────────────> 读取数据         │
  │                                    │ <───────────────                  │
  │ <─────────────── 代理返回数据       │                                   │
  │                                    │ 记录进度：downloadedBytes += n    │
  │                                    │                                   │
  │  3. 中断后恢复                      │                                   │
  │ ───────────────> GET /download/taskId
  │                                    │ 查询进度：从50MB继续              │
  │                                    │ ───────────────> Range: 50MB-     │
```

### 数据模型

```csharp
public class DownloadTask
{
    public Guid Id { get; set; }
    public Guid FileVersionId { get; set; }
    public string ClientId { get; set; }        // 设备标识
    public long FileSize { get; set; }
    public long DownloadedBytes { get; set; }   // 已下载字节数
    public DownloadStatus Status { get; set; }
    public DateTime CreatedAt { get; set; }
    public DateTime ExpiresAt { get; set; }     // 任务过期时间
}
```

### 服务端实现

```csharp
[HttpGet("{taskId}")]
public async Task DownloadWithProgress(Guid taskId, CancellationToken cancellationToken)
{
    var task = await _dbContext.DownloadTasks.FindAsync(taskId);
    var version = await _versionService.GetVersionAsync(task.FileVersionId);
    
    // 从上次位置开始
    var startPosition = task.DownloadedBytes;
    
    using var stream = await _storageProvider.GetStreamAsync(
        version.ObjectName, cancellationToken);
    
    stream.Seek(startPosition, SeekOrigin.Begin);
    
    var buffer = new byte[81920];
    int read;
    while ((read = await stream.ReadAsync(buffer, cancellationToken)) > 0)
    {
        await Response.Body.WriteAsync(buffer.AsMemory(0, read), cancellationToken);
        
        // 更新进度
        task.DownloadedBytes += read;
        await _dbContext.SaveChangesAsync(cancellationToken);
    }
    
    task.Status = DownloadStatus.Completed;
    await _dbContext.SaveChangesAsync(cancellationToken);
}
```

### 优点

| 优点 | 说明 |
|------|------|
| 精确控制 | 服务端实时掌握下载进度 |
| 跨设备续传 | 登录后可在任意设备续传 |
| 可中断恢复 | 客户端崩溃后可精确恢复 |
| 统一权限 | 便于实现复杂的权限控制 |

### 缺点

| 缺点 | 说明 |
|------|------|
| 服务端压力大 | 所有流量经过服务端 |
| 带宽成本高 | 需要更大的出口带宽 |
| 实现复杂 | 需要维护下载任务状态 |

## 方案四：P2P分片下载（BitTorrent风格）

### 原理

将文件分成多个小块，多个客户端之间互相分享已下载的分片，减轻服务端压力。

### 流程图

```
客户端A                           Tracker服务器                    客户端B
  │                                    │                              │
  │  1. 加入下载网络                    │                              │
  │ ───────────────> 注册下载任务       │                              │
  │ <─────────────── 返回peers列表      │                              │
  │     [ClientB, ClientC, ...]        │                              │
  │                                    │                              │
  │  2. 从多个peers获取分片             │                              │
  │ ───────────────> 请求分片0          │                              │
  │ <───────────────────────────────────────────────────────────────  │
  │     从ClientB获取分片0              │                              │
  │                                    │                              │
  │ ───────────────> 请求分片1          │                              │
  │ <───────────────────────────────────────────────────────────────  │
  │     从ClientC获取分片1              │                              │
  │                                    │                              │
  │  3. 分享已下载的分片                │                              │
  │ <───────────────────────────────────────────────────────────────  │
  │     响应ClientB的分片请求           │                              │
```

### 适用场景

- 超大文件（>10GB）分发
- 多人同时下载同一文件
- 带宽受限的环境
- 开源软件分发

### 优点

| 优点 | 说明 |
|------|------|
| 高并发 | 越多人下载越快 |
| 容错性好 | 部分peer离线不影响整体 |
| 减轻源站压力 | 大部分流量在peers间传输 |

### 缺点

| 缺点 | 说明 |
|------|------|
| 实现复杂 | 需要Tracker、DHT等组件 |
| 隐私问题 | 暴露IP地址给其他人 |
| 初始种子 | 需要至少一个完整副本 |
| 不适合小文件 |  overhead 太高 |

## 方案对比总结

| 维度 | HTTP Range | 预签名URL | 服务端代理 | P2P |
|------|-----------|-----------|-----------|-----|
| **实现复杂度** | 低 | 中 | 中 | 高 |
| **服务端压力** | 低 | 极低 | 高 | 极低 |
| **跨设备续传** | ❌ | ❌ | ✅ | ✅ |
| **并发下载** | ✅ | ✅ | ❌ | ✅ |
| **权限控制** | 中 | 中 | 强 | 弱 |
| **适用文件大小** | 任意 | 任意 | <1GB | >10GB |
| **浏览器支持** | ✅ | ✅ | ✅ | ❌ |
| **移动端支持** | ✅ | ✅ | ✅ | 部分 |

## 推荐方案选择

### 场景一：Web浏览器下载（推荐方案一）

```
使用 HTTP Range 请求
- 标准协议，浏览器原生支持
- 实现简单，无需额外依赖
- 支持暂停/恢复（通过Service Worker）
```

### 场景二：移动端APP下载（推荐方案二）

```
使用 预签名URL分段下载
- 减轻服务端带宽压力
- 支持后台下载
- 可配合CDN加速
```

### 场景三：企业内网/权限严格（推荐方案三）

```
使用 服务端代理流
- 精确控制下载权限
- 可审计下载行为
- 支持跨设备续传
```

### 场景四：超大文件分发（推荐方案四）

```
使用 P2P分片下载
- 适合软件分发、视频分发
- 越多人下载越快
- 需要专门的客户端支持
```

## 实现建议

### 渐进式实现路径

```
Phase 1: HTTP Range（基础功能）
    ├── 支持 Range 请求
    ├── 客户端分片管理
    └── 本地进度存储

Phase 2: 预签名URL（优化性能）
    ├── 对接对象存储
    ├── 生成预签名URL
    └── 客户端直接使用

Phase 3: 服务端代理（增强控制）
    ├── DownloadTask 表
    ├── 进度持久化
    └── 跨设备续传

Phase 4: P2P（可选高级功能）
    ├── WebRTC 数据传输
    ├── 分片共享协议
    └── Tracker服务
```

### 关键注意事项

1. **分片大小选择**
   - 太小：HTTP请求开销大
   - 太大：续传粒度粗
   - 推荐：1MB - 10MB

2. **校验机制**
   - 每个分片独立MD5校验
   - 整文件SHA256校验
   - 发现损坏重传单个分片

3. **并发控制**
   - 浏览器限制：HTTP/1.1 6个并发，HTTP/2 100个
   - 推荐：3-5个并发分片

4. **存储空间**
   - 预留2倍文件大小的临时空间
   - 定期清理未完成的下载
